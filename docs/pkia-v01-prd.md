# PKIA v0.1 产品需求文档（PRD）

> 文档版本：v0.1-draft · 作者：PKIA Product · 日期：2026-06-14
> 本 PRD 不涉及任何数据库设计、代码架构或技术实现细节。架构设计将在 PRD 审核通过后启动。

---

## 1. 用户画像

- **身份**：全栈独立开发者 / 个人研究者
- **日常工作**：在 GitHub 上探索开源项目、阅读论文（Arxiv）、持续构建个人技术积累
- **信息消费习惯**：每天固定时间（如早晚各30分钟）扫一遍趋势和论文，但看完容易忘
- **痛点特征**：信息过载，但又不愿意完全错过重要内容；希望有"自己的判断依据"而非纯靠算法推荐

---

## 2. 核心痛点

1. **看过就忘**：上周看了一篇重要的论文，现在想不起来是哪篇、什么内容、为什么觉得重要。
2. **信息入口太宽**：Arxiv 每天几千篇、GitHub Trending 每天几十个项目——根本看不完，但漏了又不安。
3. **缺乏关联感知**：看到一篇新论文和一个新项目，可能跟之前关注的东西强相关，但自己需要手动回忆和比对。
4. **决策疲劳**：不知道今天该深度看什么、该跳过什么——每天都像重新开始。

---

## 3. 核心价值

PKIA v0.1 提供**一个最小但闭环的"个人知识过滤器"**：

> 帮你记住"什么值得关注"，并告诉你"今天的新东西跟你有什么关系"。

具体价值：
- **自动聚合** Arxiv + GitHub Trending 当日信息
- **基于历史兴趣的优先级分层**（而非固定 Top N）
- **每项推荐的"理由"**——为什么你可能关心这个
- **轻量查阅历史**——你标注过什么、看过什么

---

## 4. MVP 功能列表

| 编号 | 功能 | 说明 |
|------|------|------|
| F1 | **Arxiv 信息抓取** | 每日自动拉取指定分类（可配置 cs.AI / cs.CL / cs.LG 等）新论文 |
| F2 | **GitHub Trending 信息抓取** | 每日自动拉取 GitHub Trending（日/周榜） |
| F3 | **兴趣标注** | 用户可对论文/项目标注：`兴趣` / `待深度阅读` / `已读` / `不感兴趣` |
| F4 | **每日推荐报告** | 基于历史标注，生成分层推荐（见第7节），每日一份 |
| F5 | **推荐理由生成** | 每条推荐附带简短理由（如：*与你标注"感兴趣"的 3 篇论文同领域*） |
| F6 | **历史回顾** | 可按"我标注过什么"查询（时间/标签/来源维度） |

## 5. 非目标（Out of Scope）

- ❌ 多信息源（仅 Arxiv + GitHub Trending，不做 RSS、Twitter、HackerNews 等）
- ❌ 多用户 / 团队协作 / SaaS 化
- ❌ 知识图谱 / 三层记忆系统 / 向量检索
- ❌ 自动推送（无邮件、无社交媒体自动发布）
- ❌ MCP 集成 / Plugin 扩展机制
- ❌ 全文阅读 / PDF 解析 / 论文正文分析
- ❌ 实时流 / WebSocket 推送
- ❌ 第三方账号登录 / 权限系统
- ❌ Android / iOS App

## 6. 用户使用流程

典型的**一天**：

| 时段 | 行为 | 系统动作 |
|------|------|----------|
| 09:00 | 用户打开 PKIA | 系统已生成**今日推荐报告**（基于凌晨自动抓取的最新 Arxiv 论文 + GitHub Trending） |
| 09:00–09:15 | 浏览"强烈推荐"层（3~8条），快速决策 | 理由简洁，一眼判断是否值得深看 |
| 09:15–09:25 | 对其中 2 条标注"待深度阅读"，对其余标注"不感兴趣" | 系统记录标注，更新兴趣模型参考 |
| 09:25–09:30 | 扫一眼"可能相关"层，跳过后关闭 | — |
| 当天碎片时间 | 打开之前标注"待深度阅读"的条目，阅读原文/打开 GitHub 链接 | 系统不跟踪外部阅读行为（v0.1） |
| 22:00 | 复盘当天标注 | 查看"本周我已标注"列表，确认无遗漏 |

## 7. 输出形式设计：三层推荐

每份每日报告按三层组织，不固定条数：

### 第一层：🔴 强烈推荐（预计 3~8 条）

**入选条件**：
- 与用户历史"兴趣"/"待深度阅读"条目存在明确关联（同领域、同关键词、同作者/组织、引用关系）
- 或 GitHub Trending 评分异常高（star 增速 > 阈值）且与用户技术栈相关

**展示信息**：
- 标题、作者/仓库、来源（Arxiv / GitHub）
- 一句话摘要（论文摘要第一句，或项目 description）
- **推荐理由**：如「*与你3天前标注"兴趣"的"Retrieval-Augmented Generation Survey"出现 4 个共现关键词*」

### 第二层：🟡 可能相关（预计 5~15 条）

**入选条件**：
- 与历史兴趣存在弱关联（宽泛领域匹配）
- 或当日高频关键词（当天 Arxiv 论文中出现频次显著升高的术语）
- 或 GitHub Trending 热门但与用户领域仅部分重叠

**展示信息**：
- 标题 + 来源
- 一句话理由（如「*今日 Arxiv 高频主题：Multi-Agent Systems*」）

### 第三层：⚪ 其余新条目（可折叠，默认不展开）

**入选条件**：
- 当日所有未在前两层的条目
- 供用户按需浏览，保持"不遗漏"的安全感

**展示信息**：
- 仅标题列表 + 来源标签

---

**与历史兴趣的关联机制**（v0.1 简化版）：

- 系统维护一个"兴趣种子集"：用户所有标注为"兴趣"和"待深度阅读"的历史条目
- 新条目与种子集做**关键词匹配**（标题/摘要/description 的 TF-IDF 或简单共现）
- 匹配分数决定入层优先级
- 种子集随时间衰减（用户超过 30 天未交互的方向权重降低）

## 8. 成功标准

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 每日使用留存 | 连续使用 ≥ 5 天/周 | 启动次数统计 |
| 标注参与率 | 每天至少对 ≥ 1 条内容标注 | 标注行为记录 |
| 推荐有效点击率 | 第一、二层推荐中，用户点击"查看原文/项目"的比例 ≥ 30% | 交互埋点 |
| "安全感"主观反馈 | 用户感觉"没有错过重要信息" | 每两周简单主观评分（1–5） |

**v0.1 成功的定性判断**：用户一周后仍主动打开 PKIA，且能说出"它帮我省了时间"。
